**Phase**: 1 | **Date**: 2026-04-28 | **Plan**: [plan.md](./plan.md)

## New Entity: `Center`

**Table**: `centers`

| Column | Type | Constraints | Notes |
|--------|------|-------------|-------|
| `id` | UUID | PK, NOT NULL | Auto-generated |
| `name` | String(200) | NOT NULL, UNIQUE | Center display name |
| `code` | String(20) | NOT NULL, UNIQUE | Human-readable ID (e.g., `CTR-001`); auto-generated sequential |
| `registered_by_id` | UUID | FK → `users.id`, nullable | Superadmin who created the center |
| `is_active` | Boolean | NOT NULL, default `true` | Deactivated centers block all center logins |
| `created_at` | TIMESTAMPTZ | NOT NULL | Auto-managed |
| `updated_at` | TIMESTAMPTZ | NOT NULL | Auto-managed |

**Relationships**:
- `Center` → `User` (registered_by): many-to-one, nullable
- `Center` ← `User` (members): one-to-many (all non-superadmin users belong to a center)
- `Center` ← `Student`, `Teacher`, `ClassSession`, etc.: one-to-many (all tenant-scoped entities)

**Validation**:
- `name`: 1–200 characters, stripped of leading/trailing whitespace, UNIQUE case-insensitive check at application layer before insert
- `code`: auto-generated by the system using format `CTR-{NNN}` (three-digit zero-padded sequential counter); not user-editable

## Modified Entity: `User`

**Change type**: ADD COLUMN

| Column | Type | Change | Notes |
|--------|------|--------|-------|
| `role` | String(20) | Extended values | Now accepts: `superadmin`, `admin`, `staff`, `parent` |
| `center_id` | UUID | NEW FK → `centers.id`, nullable | NULL only for `superadmin` role; NOT NULL for all other roles |

**Constraint**: CHECK `(role = 'superadmin' AND center_id IS NULL) OR (role != 'superadmin' AND center_id IS NOT NULL)`

**Existing rows**: All current users assigned `center_id = <legacy-center-id>` (the auto-created `CTR-001` center in migration).

## Modified Entities: All Tenant-Scoped Tables

The following tables receive a new `center_id UUID NOT NULL FK → centers.id` column:

| Table | Existing FK chain | Notes |
|-------|------------------|-------|
| `students` | — | Direct tenant root |
| `teachers` | — | Direct tenant root |
| `class_sessions` | — | Direct tenant root |
| `class_enrollments` | → `class_sessions` | `center_id` added for fast filter; must match class's center |
| `packages` | → `class_sessions`, `students` | `center_id` added for fast filter |
| `payment_records` | → `packages` | `center_id` denormalized for fast filter |
| `attendance` | → `class_sessions`, `students` | `center_id` denormalized for fast filter |
| `renewal_reminders` | → `packages` | `center_id` denormalized for fast filter |
| `lesson_kinds` | — | Each center maintains its own lesson kind vocabulary |
| `student_status_history` | → `students` | `center_id` denormalized for fast filter |

**Denormalization rationale**: Adding `center_id` directly to every table avoids multi-join paths to derive the owning center, which is necessary for every single query to enforce isolation.

## Alembic Migration

**File**: `backend/alembic/versions/013_multi_tenant_centers.py`

**Operations (in order)**:
1. CREATE TABLE `centers` with all columns
2. INSERT legacy center row: `{id: <uuid>, name: 'Legacy Center', code: 'CTR-001', is_active: true}`
3. For each tenant table:
   a. ADD COLUMN `center_id UUID NULLABLE`
   b. UPDATE all rows `SET center_id = <legacy-center-id>`
   c. SET NOT NULL on `center_id`
   d. ADD FK CONSTRAINT `REFERENCES centers(id) ON DELETE RESTRICT`
   e. CREATE INDEX on `center_id`
4. ADD COLUMN `center_id UUID NULLABLE` to `users`; UPDATE all existing users to legacy center; set role-conditional NOT NULL (nullable for superadmin, others constrained at app layer)
5. INSERT superadmin user: `{username: 'superadmin', role: 'superadmin', center_id: NULL, ...}`

## CRUD / Service Layer Contract

All CRUD functions in `backend/app/crud/` that query tenant-scoped tables MUST accept and apply a `center_id: UUID` parameter. The pattern:

```
# Every list/get function MUST include:
WHERE <table>.center_id = :center_id
```

The `center_id` is extracted from the authenticated user in the API layer and injected into all CRUD calls. Functions MUST NOT provide a default that bypasses filtering (no `center_id=None` defaults).

The only exemptions are:
- `crud/center.py` functions (operate on the `centers` table itself, superadmin only)
- `crud/user.py` functions used by auth middleware (must be able to look up any user for authentication)
